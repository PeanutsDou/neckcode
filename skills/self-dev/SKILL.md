# Self-Dev — 通用开发流程管理

适用于所有项目的标准化开发流程。**核心原则：文档先行，确认后开发，完成后同步。**

## 核心规则

**绝对禁止在开发文档未经用户确认的情况下直接进行开发落地。**

## 开发状态

| 状态 | 含义 |
|------|------|
| 规划中 | 需求已记录，方案讨论中，待用户确认 |
| 执行中 | 方案已确认，开发进行中 |
| 已完成 | 开发完成并验证，待归档 |

## 文档命名规范

```
<项目根目录>/spec/<功能子目录>/<功能>-<状态>-<日期>.md
```

- 状态使用英文：planning / in-progress / completed
- 日期格式：YYYYMMDD

示例：
- `spec/agent-panel/agent-panel-planning-20260514.md`
- `spec/ui-refactor/sidebar-in-progress-20260520.md`
- `spec/bug-fix/memory-leak-completed-20260510.md`

## 开发文档标准结构

所有规划文档必须遵循以下结构，确保信息完整、可执行、可追踪。

### 一、需求概述（元信息区）

开篇用 blockquote 概括**核心目标 + 一句话方法论**：

> **核心目标**：一句话说清楚这个功能要解决什么问题。
> **关键思路**：一句话说清楚怎么解决。

### 二、架构设计

ASCII 架构图 + 模块职责描述，展示：

- 功能与现有系统的关系
- 各子模块的输入/输出
- 数据流向
- 职责边界（什么归这个模块、什么不归）

### 三、功能分解

表格化列出所有子功能：

| # | 模块 | 作用 | 依赖 | 状态 |
|---|------|------|------|------|
| 1 | xxx | xxx | — | ☐ 未开始 |

### 四、技术方案

| 维度 | 说明 |
|------|------|
| 数据存储 | 数据结构、存储位置 |
| UI 组件 | 新增/修改的组件列表 |
| IPC 通道 | 新增的 main-renderer 通信通道 |
| 运行时 | 核心逻辑变更 |

### 五、影响范围

列出受影响的文件和模块，评估改动量。

### 六、执行规划（Phase 编排）

按依赖关系分 Phase，每个 Phase 包含：

- **准入条件**：进入该 Phase 的前提
- **任务列表**：具体开发任务
- **产物**：该 Phase 完成后输出什么
- **验证方式**：如何确认该 Phase 正确完成

### 七、测试计划

分层验证（L1 冒烟 → L2 单元 → L3 集成 → L4 端到端）：

| 层级 | 测试内容 | 预期结果 | 不通过说明 |
|------|----------|----------|------------|
| L1 | xxx | xxx | 可能原因 |

### 八、环境与约束

涉及的外部依赖、版本要求、路径约束。

### 九、已确认项 / 待确认项

- [x] 已确认的决策
- [ ] 待确认的问题

## 工作流程

### Step 1：接收需求

```
a. 检查 spec/ 目录下是否存在相关子目录和文档
b. 如存在相关文档 → 读取并告知用户现有内容，询问是否继续
c. 如不存在 → 创建子目录和规划文档，子目录名以需求关键词命名
d. 文档初始状态设为「规划中」
e. 按标准结构撰写文档，待确认项明确标出
```

### Step 2：需求细化

```
a. 在文档中详细记录需求描述、技术方案、影响范围、风险点
b. 用户确认后 → 状态改为「执行中」，待确认项转为已确认
c. 仅在用户明确说"开始开发"后才进入开发
```

### Step 3：开发执行

```
a. 严格按文档中确认的方案和 Phase 顺序执行
b. 如有偏离，先更新文档再继续
c. 每个 Phase 完成后更新文档中的状态和产物
```

### Step 4：开发完成

```
a. 状态改为「已完成」，更新完成日期
b. 将完成项同步到 spec/now-function-completed.md
c. 提交代码 + 文档，推送到 GitHub
```

## spec 目录结构

```
spec/
├── now-function-completed.md        # 已完成功能总览
├── <feature-a>/                     # 功能 A
│   ├── xxx-planning-YYYYMMDD.md
│   └── xxx-in-progress-YYYYMMDD.md
├── <feature-b>/                     # 功能 B
└── ...
```

## 重要提醒

- 文档是开发的唯一依据，口头需求也必须先落地为文档
- 每次完成开发后**立即更新文档**，确保反映真实进度
- spec 目录纳入 Git 版本管理，与代码同步提交
