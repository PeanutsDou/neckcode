# Self-Dev — 通用项目开发管理

## 核心铁律

**绝对禁止在开发文档未确认的情况下直接进行开发落地。**

文档是开发的唯一依据。口头需求必须落成文档并确认后才能动手。

## 开发状态

| 状态 | 英文标识 | 含义 |
|------|----------|------|
| 规划中 | `planning` | 需求已记录，方案待细化/确认 |
| 执行中 | `in-progress` | 方案已确认，正在开发 |
| 已完成 | `completed` | 开发完成，待归档到 now-function-completed.md |
| 已搁置 | `paused` | 暂时搁置，保留文档 |

## 文档命名规范

```
<项目根>/spec/<子目录>/<功能描述>-<状态>-<日期>.md
```

示例：
- `spec/agent-function/agent-fuction-planning-20260514.md`
- `spec/ui-refactor/ui-refactor-in-progress-20260520.md`
- `spec/bug-fix/context-leak-completed-20260510.md`
- `spec/new-feature/auto-save-planning-20260601.md`

规则：
- **子目录名**：按功能模块或需求范围命名，如用户未明确可先创建再确认
- **文件名**：简要描述功能 + 状态 + 创建日期（YYYYMMDD）
- **每个子目录可包含多个文档**，方便同一模块的不同阶段分别管理

## spec 目录结构

```
<项目根>/spec/
├── now-function-completed.md          # 已完成功能总览（每次完成条目后立即更新）
├── <模块A>/                           # 按需求/模块划分子目录
│   ├── xxx-planning-20260514.md
│   └── yyy-in-progress-20260520.md
├── <模块B>/
│   └── zzz-planning-20260514.md
└── ...
```

## 工作流程

### 第一步：勘察

```
1. 检查当前项目的 spec/ 目录是否存在
2. 如不存在，创建 spec/ 目录和 now-function-completed.md
3. 如 now-function-completed.md 不存在，基于现有代码生成
4. 搜索是否已有与用户需求相关的子目录或文档
5. 如有相关文档，读取内容，告知用户现状
6. 如无，告知用户，准备新建
```

### 第二步：建文档

```
1. 判断用户是否明确了需求范围/模块名
   - 明确 → 直接在 spec/<子目录>/ 下新建文档
   - 不明确 → 先提出子目录命名建议，找用户确认后再建
2. 文档状态初始设为「规划中」
3. 在文档中记录：
   - 需求描述
   - 方案讨论（待填充）
   - 影响范围
   - 待确认事项
```

### 第三步：需求讨论

```
1. 与用户充分讨论需求细节和方案
2. 将结论写入文档，更新待确认事项
3. 用户最终确认后，状态改为「执行中」
4. 此时仍不能开发，等待用户明确下达开发指令
```

### 第四步：开发执行

```
1. 仅在用户明确说「开始开发」「执行」「做吧」等指令后动手
2. 严格按照文档中确认的方案执行
3. 开发中如有偏离，先更新文档再继续
4. 阶段性完成，更新文档进度
```

### 第五步：完成后归档

```
1. 文档状态改为「已完成」，补上完成日期
2. 将完成的功能条目同步到 spec/now-function-completed.md
3. 提交代码 + 文档
```

## 适用场景

本 skill 适用于**所有项目**的开发需求，包括但不限于：

- DeepSeek Code 自身功能开发
- 用户的任何其他编程项目
- Bug 修复
- 重构
- 新工具/脚本开发
- 技能（Skill）开发

## 重要

- 只要用户提出开发需求，本 skill 就是入口
- `spec/` 目录纳入 Git，与代码一起版本管理
- 每次开发完成后**立即同步文档**，保证文档始终反映真实进度
