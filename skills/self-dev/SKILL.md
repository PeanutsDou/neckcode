# Self-Dev — DeepSeek Code 自身开发管理

## 核心规则

**绝对禁止在开发文档未确认的情况下直接进行开发落地任务。**

## 开发状态

| 状态 | 含义 |
|------|------|
| 规划中 | 需求已记录，待细化/确认 |
| 执行中 | 需求已确认，正在开发 |
| 已完成 | 开发完成，待归档 |

## 文档命名规范

```
spec/<项目子目录>/<功能>-<模块>-<状态>-<日期>.md
```

示例：
- `spec/agent-function/agent-fuction-planning-20260514.md`
- `spec/ui-refactor/ui-refactor-in-progress-20260520.md`
- `spec/bug-fix/context-leak-completed-20260510.md`

## 工作流程

### 1. 用户提出开发需求

收到开发需求后，执行以下步骤：

```
a. 检查 spec/ 目录下是否存在相关子目录和文档
b. 如有相关文档，读取并告知用户现有内容，询问是否基于现有内容继续
c. 如无相关文档：
   - 如果用户已明确需求范围，在 spec/ 下创建子目录和规划文档
   - 如果用户未明确，先创建目录结构，再找用户确认子目录命名
d. 文档状态初始设为「规划中」
```

### 2. 需求讨论与确认

```
a. 在文档中详细记录需求、方案、影响范围
b. 用户确认后，将状态改为「执行中」
c. 仅在用户明确下达开发指令后，才开始开发
```

### 3. 开发执行

```
a. 严格按照文档中确认的方案执行
b. 开发过程中如有偏离，先更新文档再继续
c. 阶段性完成后，更新文档中的进度
```

### 4. 开发完成

```
a. 将文档状态改为「已完成」
b. 更新完成日期
c. 将已完成的功能条目同步到 spec/now-function-completed.md
d. 提交代码 + 文档，推送到 GitHub
```

## spec 目录结构

```
spec/
├── now-function-completed.md          # 已完成功能总览（每次完成后更新）
├── agent-function/                    # Agent 功能相关
│   └── agent-fuction-planning-*.md
├── ui-refactor/                       # UI 重构相关（示例）
├── bug-fix/                           # Bug 修复相关（示例）
└── ...
```

每个子目录下可以有多个文档，按日期和状态区分。

## 重要提醒

- 文档是开发的唯一依据，口头需求也必须先落地为文档
- 每次基于文档完成开发后，**立即更新文档内容**，确保文档反映真实开发进度
- `spec/` 目录和 `skills/self-dev/` 均纳入 Git 版本管理
